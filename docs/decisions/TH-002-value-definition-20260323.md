# Phase 2: 価値定義（軽量版） — Supabase統合（v1）

## 日付
2026-03-23

## 議論の参加者
- AIPO, AI PM

---

## v0からの学び（仮説検証結果）

v0のドッグフーディングで得られた知見を、v1の価値定義の入力として整理する。

### 検証済みの仮説

| 仮説 | 結果 | 根拠 |
|---|---|---|
| 30秒以内に全テーマの現在地を把握できる | 検証済み | テーマ一覧+フェーズ進行インジケーターで実現 |
| mdファイルを直接読まずに判断経緯を理解できる | 検証済み | タイムラインタブで本文が閲覧可能 |
| 閲覧のみで安心感が得られるか（R4） | 部分的 | 状況把握は可能だが、「判断を下す」行為がダッシュボード外（md直接編集 or 口頭）に残っている |

### v0で顕在化した課題

1. **Go/No-Go操作のフリクション**: `awaiting-review` のテーマを確認した後、ダッシュボード上で承認/差し戻しができない。別の手段（Slack、口頭、md直接編集）で判断を伝える必要があり、ダッシュボードが「見るだけの道具」に留まっている。
2. **データ更新の遅延**: SSGのため、AIPOがmdファイルを更新してもVercelの再ビルドが走るまでダッシュボードに反映されない。リアルタイム性の欠如は「動いている証拠」原則を損なう。
3. **アクセス制御の不在**: デプロイ済みURLを知る人は誰でもアクセス可能。現時点ではドッグフーディング用途で問題ないが、Go/No-Go操作を追加する時点でオーナー認証は必須となる。
4. **判断ポイントの抽出**: タイムラインタブで全文を読む必要があり、テーマ数が増えると「どの判断が重要か」の選別に時間がかかる。S-008（判断ポイントタブ）のShouldがMustに昇格すべきタイミング。

## v1で追加する各機能が解くユーザー課題

### 1. Supabase基盤（PostgreSQL + Realtime）

| 項目 | 内容 |
|---|---|
| 解く課題 | mdファイルパースに依存するデータ取得は、ビルド時にしか更新されず、リアルタイム性がない。また、Go/No-Go操作の結果をmdファイルに書き戻すのは設計上不自然 |
| なぜ今必要か | Go/No-Go操作と認証の基盤として、永続化レイヤーが必須。mdファイルからDBへの移行なしに、v1の他の機能はすべて実装不可能 |
| 成功指標 | データソースがSupabase PostgreSQLに一元化され、既存UIが無変更で動作する（ファサードパターンの恩恵） |

### 2. SSG → ISR移行

| 項目 | 内容 |
|---|---|
| 解く課題 | SSGではデータ更新にフルビルドが必要。AIPOがテーマを進行させても、オーナーがダッシュボードで確認するまでに分単位のラグがある |
| なぜ今必要か | Supabase導入でデータソースが動的になる以上、SSGの前提が崩れる。ISRはSupabaseとの組み合わせで最小コストの移行手段 |
| 成功指標 | Go/No-Go操作後、revalidateによりページが数秒以内に最新データを反映する |

### 3. 認証（Google OIDC）

| 項目 | 内容 |
|---|---|
| 解く課題 | Go/No-Go操作を追加する以上、「誰が操作したか」の記録と「権限のない人が操作できない」保証が必要 |
| なぜ今必要か | v0は閲覧のみのため認証不要だったが、v1で状態変更操作が入る時点でアクセス制御は最低限必要。オーナー1人の運用なのでGoogle OIDC 1つで十分 |
| 成功指標 | 未認証ユーザーはGo/No-Go操作ができない。認証済みオーナーのみ操作可能 |

### 4. Go/No-Go操作

| 項目 | 内容 |
|---|---|
| 解く課題 | v0最大の未解決課題。オーナーが判断を下す行為がダッシュボード外に残っており、ワークフローが分断されている |
| なぜ今必要か | これがv1の中核価値。ダッシュボードを「見るだけの道具」から「判断を下す場所」に昇格させる。オーナーの意思決定ループを完結させる |
| 成功指標 | `awaiting-review` テーマに対して、承認/差し戻しの2アクションがダッシュボード上で完結する |

### 5. S-008 判断ポイントタブ

| 項目 | 内容 |
|---|---|
| 解く課題 | 全文を読まないと重要な判断ポイントが分からない。テーマ数とフェーズ数が増えると、タイムラインのスクロール量が増大する |
| なぜ今必要か | v0ではShouldだったが、テーマ数の増加に伴い「全文を読む時間がない」問題が顕在化。Go/No-Go判断の効率を上げるために、判断ポイントの抽出表示が重要になる |
| 成功指標 | 決定事項、却下した案、リスクが構造化されて表示され、オーナーがGo/No-Go判断に必要な情報を30秒以内に把握できる |

## v1スコープの妥当性

### 含めるべきもの（確定）

| 機能 | 理由 |
|---|---|
| Supabase基盤 | 全機能の前提条件 |
| SSG → ISR | Supabase導入に伴う必然的な変更 |
| 認証（Google OIDC） | Go/No-Go操作のアクセス制御 |
| Go/No-Go操作 | v1の中核価値 |
| S-008 判断ポイントタブ | v0でShouldだった機能のMust昇格 |

### 含めないもの（v2以降）

| 機能 | 理由 |
|---|---|
| Realtime（WebSocket購読） | ISR + revalidateで十分。リアルタイム要件はv2で検討 |
| フィルタ/ソート | テーマ数がまだ少ない。テーマ10件超で検討 |
| Slack/メール通知 | v2スコープとして定義済み |
| GitHub連携 | v2スコープとして定義済み |
| RLS（Row Level Security）の複雑な設計 | オーナー1人の運用。認証=全権限でv1は十分 |
| モバイル最適化 | Desktop-firstの方針を継続 |
| 操作履歴/監査ログ | v1では「誰が承認したか」をDB列に記録するだけで十分。詳細な監査ログはv2 |

### スコープの判断基準（v0から継承・更新）

1. Go/No-Go判断がダッシュボード上で完結するために必要か？ → Noならv2
2. オーナー1人の運用で本当に必要か？ → Noならv2
3. mdファイル→DB移行の最小構成に含まれるか？ → Noなら後回し

## リスク・不確実性

1. **R10: SSG→ISR移行のVercelランタイム変更（影響度: 高, 確率: 中）**: `output: "export"` 削除でServerless Functionsに移行。ビルド・デプロイフローが変わる。対策: Wave 1でISR移行を先に完了させ、動作確認してから認証・操作を追加。
2. **R11: Next.js 15 async cookiesとSupabase Auth SSR（影響度: 高, 確率: 中）**: `cookies()` がasyncになったことで、Supabase SSRのクライアント初期化に影響する可能性。対策: 公式ドキュメントに従い、`@supabase/ssr` の最新版を使用。
3. **R12: mdファイル→DB移行の完全性（影響度: 中, 確率: 低）**: 既存のmdファイルデータをPostgreSQLに正確に移行できるか。対策: マイグレーションスクリプトを作成し、移行前後でデータ件数・内容を検証。
4. **R13: Go/No-Go操作の楽観的更新（影響度: 中, 確率: 中）**: Server Action実行後のUI更新タイミング。対策: `revalidatePath` で即座にISRキャッシュを無効化。
5. **R14: 判断ポイント抽出のセクション見出し依存（影響度: 低, 確率: 中）**: v0のS-008と同一リスク。AIPOの出力テンプレートとの整合が前提。対策: フォールバックUI付き。
