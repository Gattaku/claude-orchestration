## 議論の経緯

### 論点1: Storyの粒度と優先順位

- **AI PMの意見**: 「判断ワークフロー改善」を最優先とし、ユーザーが最も頻繁に行う「確認待ちテーマの承認/差し戻し」の体験を集中的に改善すべき。次に「情報の見つけやすさ」、最後に「基盤改善」の順。
- **AI PDの意見**: 同意するが、「判断ポイントタブ」の実装と「サマリーカード」は同時にやるべき。判断ポイントタブがあっても、テーマ一覧からどのテーマを見るべきか判断する手がかりがないと、結局使いづらい。
- **AI Devの意見**: N+1クエリ修正は他のすべての改善の前提として最初にやるべき。テーマが増えるとページの読み込みが遅くなり、他の改善の効果を損なう。
- **議論のポイント**: AI Devの指摘を受けて、パフォーマンス改善（N+1クエリ修正）を最初のStoryとし、その後にUX改善Storyを順次実行する方針で合意。

### 論点2: 「判断ポイントタブ」の内容

- **AI PMの意見**: 各Phase完了時の「決定事項」「判断理由」「却下した案」をコンパクトに表示すべき。人間オーナーが「なぜこうなったか」を30秒で理解できるようにする。
- **AI PDの意見**: カード形式で表示し、各カードに「決定事項（太字）」「判断理由（折りたたみ）」「却下案（折りたたみ）」を配置する。全文表示は不要で、要点だけ見える設計にする。
- **AI Devの意見**: 現状のbody_htmlにはMarkdownの全文がHTMLとして入っている。決定事項だけを抽出するには、Markdownの構造を解析するか、DBスキーマに「summary」カラムを追加する必要がある。後者の方が明確。
- **議論のポイント**: DB設計の変更が必要になるため、Phase 4（技術設計）が必要との判断に至ったが、今回は計画策定のみなので、実装時に検討する。

### 論点3: 議論ログ機能の実装範囲

- **AI PMの意見**: 議論ログはこのプロダクトの差別化ポイント。ただし、MVP段階では全文表示ではなく、「誰が何を言ったか」のサマリーだけで十分。
- **AI PDの意見**: チャット風UIは実装コストが高い。まずはシンプルなリスト形式（タイムスタンプ + ロール + メッセージ要約）で始め、後からリッチにする。
- **AI Devの意見**: discussion_logsテーブルの新規作成が必要。スキーマは以下を提案:
  - `id`, `theme_id`, `agent_role`, `direction` (request/response), `message`, `created_at`
  - Supabase Realtimeを有効にして、議論が進行中にリアルタイムで表示を更新できるようにする
- **議論のポイント**: 議論ログは独立したテーマ（TH-004相当）として分離し、今回のUX改善ロードマップの中では後半に位置づけることで合意。

## アップデート計画（優先度付きStory一覧）

### 優先度: P0（即座に着手）

#### S-301: Supabaseクエリ最適化
- **タイトル**: テーマ一覧・詳細のデータ取得を最適化する
- **ユーザーストーリー**: ダッシュボードユーザーとして、テーマが増えても素早くページが表示されてほしい。なぜなら、毎朝の確認作業を短時間で終わらせたいから。
- **受け入れ条件**:
  - [ ] `getAllThemes()` のN+1クエリを解消し、JOINまたはバッチクエリで1-2回のクエリに収める
  - [ ] `getThemeById()` が全テーマを取得せず、対象テーマだけを直接クエリする
  - [ ] テーマ10件以上のテストデータで、レスポンスタイムが500ms以下であること
- **影響範囲**: `src/lib/data/themes.ts`
- **規模**: S
- **期待効果**: ページ読み込み速度の大幅改善、今後のスケーラビリティ確保

### 優先度: P1（P0完了後に着手）

#### S-302: ダッシュボードサマリーカード
- **タイトル**: テーマ一覧ページにサマリー統計カードを追加する
- **ユーザーストーリー**: オーナーとして、ダッシュボードを開いた瞬間に全体の状況を把握したい。なぜなら、今日対応が必要なことがあるかどうかを瞬時に判断したいから。
- **受け入れ条件**:
  - [ ] テーマ一覧上部に統計カードが表示される（全テーマ数、進行中、確認待ち、完了）
  - [ ] 確認待ちの数は目立つ色（アンバー）でハイライトされる
  - [ ] カードをクリックすると対応するステータスでフィルタされる
- **影響範囲**: `src/app/page.tsx`, 新規コンポーネント `summary-cards.tsx`
- **規模**: S
- **期待効果**: 日次チェックの所要時間短縮、対応必要事項の見逃し防止

#### S-303: テーマ一覧のフィルタリングとソート
- **タイトル**: テーマ一覧にステータスフィルタとソート機能を追加する
- **ユーザーストーリー**: オーナーとして、特定のステータスのテーマだけを表示したい。なぜなら、テーマが増えてきたときに目的のテーマを素早く見つけたいから。
- **受け入れ条件**:
  - [ ] ステータス（進行中/確認待ち/完了/保留）でフィルタリングできる
  - [ ] 更新日時順/作成日時順でソートを切り替えられる
  - [ ] フィルタ状態がURLクエリパラメータに反映され、共有可能である
  - [ ] フィルタ結果が0件の場合、適切なEmptyStateが表示される
- **影響範囲**: `src/components/theme-list.tsx`, 新規コンポーネント `theme-filters.tsx`
- **規模**: S
- **期待効果**: テーマ数増加時の操作性維持

#### S-304: 判断ポイントタブの実装
- **タイトル**: テーマ詳細の「判断ポイント」タブを実装する
- **ユーザーストーリー**: オーナーとして、各フェーズでAIチームがどんな判断を下したかを一目で把握したい。なぜなら、Go/No-Goの判断をする前に、議論の結論と根拠を効率よく確認したいから。
- **受け入れ条件**:
  - [ ] テーマ詳細の「判断ポイント」タブに、各フェーズの判断サマリーが表示される
  - [ ] 各判断カードに「決定事項」「判断理由」が表示される
  - [ ] 「却下した案」は折りたたみで表示され、必要に応じて展開できる
  - [ ] フェーズごとに時系列で並び、進行状況が視覚的にわかる
- **影響範囲**: `src/components/timeline-tab.tsx`, 新規コンポーネント `decision-card.tsx`
- **規模**: M（DBスキーマへのsummaryカラム追加を含む場合はM、body_htmlのパースで対応する場合はS）
- **期待効果**: Go/No-Go判断の精度向上、判断に要する時間の短縮

## ロードマップ概要

```
Phase 1: 基盤改善（1-2日）
  S-301: Supabaseクエリ最適化 [P0]

Phase 2: 判断ワークフロー改善（3-5日）
  S-302: サマリーカード [P1]
  S-303: フィルタ・ソート [P1]
  S-304: 判断ポイントタブ [P1]

Phase 3: 操作性改善（3-5日）
  S-305: モバイル対応 [P2]
  S-306: 確認待ちバナー詳細化 [P2]

Phase 4: 体験強化（5-10日）
  S-307: ダークモード [P3]
  S-308: 議論ログ機能 [P3]
  S-309: リアルタイム更新 [P3]
```

## 決定事項
- 上記9つのStoryを優先度順に段階実装する
- 各StoryはExpress Path（S規模）またはStandard Path（M/L規模）で個別テーマとして管理する
- P0→P1→P2の順に実装し、各Phase完了後にユーザーフィードバックを収集する
- P3は中期的なバックログとし、P2完了後に再評価する

## 判断理由
- ユーザーの「使いづらい」を「判断に必要な情報へのアクセスの遅さ」と定義し、判断ワークフローの改善を最優先とした
- パフォーマンス改善（S-301）を最初に置いたのは、他のすべての改善の体感効果を最大化するため
- 議論ログ機能（S-308）を後半に置いたのは、実装コストが高く、現時点でのユーザー数が限定的なため

## 却下した案とその理由
- **全面リニューアル案**: 現在のプロダクト規模（テーマ2件、ユーザー1名）に対してオーバーエンジニアリング
- **通知機能（メール/Slack連携）**: MVP段階では過剰。リアルタイム更新で代替可能
- **マルチユーザー対応（ロール別権限）**: 現時点ではオーナー1名の想定であり不要

## リスク・不確実性
- ユーザーの「使いづらい」の具体的な指摘が不明なため、P1完了後のフィードバックで方向修正が必要になる可能性がある
- 判断ポイントタブ（S-304）はbody_htmlからの情報抽出方法次第でDBスキーマ変更が必要になりうる
- 議論ログ機能（S-308）のスキーマ設計はまだ未確定

## 次のアクション
- オーナーのレビュー後、P0（S-301）から実装を開始する
- 各StoryをGitHub Issueまたは個別テーマとして登録する
