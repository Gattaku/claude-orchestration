# Phase 0: トリアージ — リアルタイム進捗Push基盤

## 日付
2026-03-25

## トリアージ結果
- Input種別: B（オーナー直接指示）
- Input内容: ローカルClaude Codeから議論の中身をその都度SupabaseにPushし、Vercelダッシュボードでリアルタイムに進捗を把握したい
- 規模: S
- パス: Express
- スキップするPhase: Phase 1（インサイト抽出）, Phase 2（価値定義）, Phase 3（Story策定）, Phase 4（技術設計）
- スキップ理由:
  - Phase 1: ユーザーリサーチではなくオーナー直接指示のため不要
  - Phase 2: オーナーの課題が明確（「進捗をリアルタイムで把握したい」）のため不要
  - Phase 3: 簡易Storyで十分な規模
  - Phase 4: 既存技術スタック内で完結、新技術導入不要

## 議論の経緯

### ラウンド1: 価値提供の方向性（AI PM vs AI PD）

**AI PMの主張**: 「本質的課題は『キックが面倒』ではなく『議論中に何が起きているか見えない』こと。まずSupabase Realtimeで議論ログのライブ表示に絞るべき。Webからのキックは後回し。」

**AI PDの主張**: 「最も重要な体験改善はダッシュボード上で議論をキックできるようにすること。ターミナルとの往復を解消することが価値。」

**AI Devの提案**: ハイブリッド方式（議論フェーズはClaude API、実装フェーズはローカルCLI）を推奨。3〜5日で実装可能。

### ラウンド2: 対立の解消

**AI PM（修正後）**: AI PDの「キック + ポーリング」アプローチに同意。ただしキック機能単体では不十分、ログ可視化とセットでMVPが成立する条件を付与。

**AI PD（修正後）**: AI PMの「見守る体験が本質」に同意。「レフェリーの仕事は見ることであり、笛を吹くことではない」。キックと可視化の両方をMVPに含めるべき。

### オーナーによる軌道修正

「ローカルのClaude Codeを使い続けたい。Vercel上のSupabaseにリアルタイムで状況を伝えられるだけでも価値がある。」

→ Claude API方式は不要。既存のローカルCLIからSupabaseにPushする仕組みの改善に方向転換。

### ラウンド3: 軌道修正後の再検討

**AI PM**: 「仕組みの大部分は既にある。『動かない仕組み』を『動く仕組み』にするだけで体験は大きく変わる。規模Sで実現可能。」

**AI Dev**: 「環境変数修正 + revalidate短縮 + Phase進捗Pushスクリプト追加が費用対効果の高いMVP。」

## 検討した選択肢

### 案A: Claude API直接呼び出し（Webからキック）
- 内容: Vercel ServerlessからClaude APIを呼び出し、ローカルCLIを使わない
- メリット: ターミナル不要の完全Web体験
- デメリット: APIコスト、タイムアウト制約、プロンプト整合性リスク、オーナーがローカルCLI継続を希望

### 案B: 既存Push仕組みの修正 + revalidate短縮（採用）
- 内容: .env.local修正、dotenvの読み込み修正、revalidate=10、Phase進捗Pushスクリプト追加
- メリット: 最小変更で即効果、既存パターンの延長、リスク極小
- デメリット: リアルタイム性は最大10秒遅延（十分実用的）

### 案C: Supabase Realtime導入
- 内容: discussion_logsのINSERTをWebSocketでリアルタイム表示
- メリット: 真のリアルタイム性
- デメリット: Client Component化が必要、現時点では過剰

## 決定事項
- 案Bを採用：既存Push仕組みの修正 + revalidate短縮 + Phase進捗Pushスクリプト追加
- 案A, Cは将来課題として記録

## 判断理由
- オーナーがローカルClaude Code継続を明示したため、案Aは不適
- 既存の仕組み（insert-discussion-log.ts）が環境変数未設定で動いていなかっただけ。修正すれば即効果
- revalidate=10で体感的に十分なリアルタイム性が得られる
- 最小変更で最大効果のMVP思考に合致

## 却下した案とその理由
- 案A（Claude API直接呼び出し）: オーナーがローカルCLI継続を希望。将来の拡張候補として記録
- 案C（Supabase Realtime）: revalidate=10で十分。ペインポイントが明確化してから投入

## リスク・不確実性
- AIPOがBashコマンド実行を忘れる可能性（繰り返すならClaude Code Hooks導入を検討）
- revalidate=10でも遅いと感じる場合、Supabase Realtimeに昇格
