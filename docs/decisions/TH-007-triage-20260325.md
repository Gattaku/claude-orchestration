## 議論の経緯

### 論点1: 修正アプローチの選定

**AIPOからAI Devへの依頼内容**:
- 3つの修正案（hooks自動記録、サブエージェント自己記録、ハイブリッド）の技術的実現可能性調査
- Claude Code PostToolUse hookのデータ構造・仕様確認
- theme-id自動抽出の方法検討

**AI Devの調査結果**:

PostToolUse hookの仕様として、stdinで以下のJSONを受け取ることを確認：
```json
{
  "tool_name": "Agent",
  "tool_input": { "prompt": "...", "subagent_type": "ai-pm" },
  "tool_response": "..."
}
```
これにより `tool_input.prompt` からTH-XXXを正規表現で抽出可能、`tool_input.subagent_type` でエージェント種別を特定可能と判明。

### 論点2: theme-id特定方法の比較

AI Devが3つの方法を比較分析：

| 方法 | 内容 | 評価 |
|------|------|------|
| **A. プロンプトから正規表現抽出** | `tool_input.prompt` から `/TH-\d{3}/` を検索 | 現実的。複数マッチ時は最初のものを採用 |
| **B. transcript.jsonlからの探索** | `transcript_path` から会話コンテキストを遡って検索 | JSONLパースが重く、現実的でない |
| **C. 環境変数で明示渡し** | `CURRENT_THEME_ID` 環境変数をセット | Claude Code会話中に環境変数を永続セットする手段がなく、実現困難 |

**AIPOの判断**: 方法Aを採用。AIPOの定義に「プロンプト先頭にTH-XXXを必ず含める」ルールを追加することで、抽出の確実性を担保。

### 論点3: hooks vs サブエージェント自己記録の比較

AI Devが7つの観点で詳細比較：

| 観点 | hooks自動記録 | サブエージェント自己記録 |
|------|-------------|----------------------|
| 確実性 | **高い** — システムレベルで強制 | 低い — LLMが指示に従うか不確実 |
| request/response | **両方記録可能** | responseのみ |
| 記録の網羅性 | **高い** — 全Agent呼び出し自動補足 | 低い — 実行忘れのリスク |
| 記録の品質 | やや低め — 全文が入る | **高め** — LLMが要約可能 |
| 実装コスト | 中 — hookスクリプト新規作成 | **低** — 既存指示の改善のみ |
| 保守コスト | **低** — 一度作れば設定のみ | 高 — プロンプト調整が継続的に必要 |
| デバッグ性 | **高い** — 実行ログ確認可能 | 低い — 記録されない場合に原因不明 |

**AI Devの推奨**: ハイブリッド案（hooks主軸 + サブエージェント自己記録をフォールバック維持）

**AIPOの判断**: AI Devの推奨を採用。根拠:
- バグの直接原因が「LLMがプロンプト指示に従わない」ことであるため、同じアプローチ（サブエージェント自己記録のみ）では解決にならない
- hookによるシステムレベル強制が本質的な解決策
- 記録品質の低さ（全文が入る）は3000文字切り詰めで許容範囲

### 論点4: hookスクリプトの実装方針

AI Devが指摘した技術的注意点：

1. **依存関係の最小化**: `supabase-js` ではなくSupabase REST APIを直接fetch呼び出しすることで、TypeScript コンパイル不要の `.mjs` で実装可能
2. **Windows環境対応**: シェルスクリプトではなくNode.jsスクリプトを使用（開発環境がWindows）
3. **.env.local読み込み**: dotenvに依存せず、自前でパースして環境変数を読み込む
4. **エラーハンドリング**: すべてのケースで `exit 0` を返し、Claude Codeの会話フローを絶対に止めない
5. **agent_roleマッピング**: `subagent_type`（`ai-pm`）→ DB enum（`AI PM`）の変換が必要
6. **タイムアウト設定**: hook設定で30秒のタイムアウトを明示

## 検討した選択肢

### 案A: PostToolUse hookによる自動記録
- 内容: Claude Code hookでAgent使用後に自動でSupabase INSERT
- メリット: システムレベル強制、request/response両方記録、記録漏れなし
- デメリット: メッセージが要約されず全文（3000文字切り詰め）

### 案B: サブエージェント自己記録の強化
- 内容: ai-pm/pd/dev定義ファイルのログ記録指示を強化
- メリット: 実装コスト低、要約された記録が残る
- デメリット: LLMがプロンプト指示に従うか不確実（バグの直接原因と同じ）

### 案C: ハイブリッド（採用）
- 内容: 案A主軸 + 案B既存フォールバック維持
- メリット: 確実性と品質のバランス
- デメリット: 重複記録の可能性（hookとサブエージェントの両方が記録する場合）

## 決定事項
- 案C（ハイブリッド）を採用。hooks主軸、サブエージェント自己記録は既存のまま維持
- hookスクリプトは `.claude/hooks/log-discussion.mjs` に純Node.js（ESM）で実装
- hook設定は `.claude/settings.json`（プロジェクト共有）に配置
- AIPO定義を更新し、プロンプトにTH-XXXを必ず含めるルールを明記
- メッセージは3000文字で切り詰め

## 判断理由
- バグの根本原因が「LLMへのプロンプト指示のみでは強制力がない」ことであるため、システムレベルでの強制（hooks）が本質的な解決策
- REST API直接呼び出しにより、TypeScript コンパイル不要で高速・軽量に動作
- exit 0の徹底により、記録失敗がプロダクト利用に一切影響しない安全設計

## 却下した案とその理由
- **案Bのみ**: バグの直接原因と同じアプローチ（プロンプト指示のみ）のため、再発リスクが高い
- **transcript.jsonlからのtheme-id探索**: JSONLパースが重く、hookの実行時間が長くなる
- **環境変数によるtheme-id受け渡し**: Claude Code会話中に環境変数を永続セットする手段がない

## リスク・不確実性
- プロンプトにTH-XXXを含め忘れた場合は記録されない（ルール明記で軽減済みだが完全ではない）
- hookスクリプトのtool_responseの構造がClaude Codeバージョンによって変わる可能性
- 3000文字切り詰めにより、長い議論の末尾が失われる可能性
